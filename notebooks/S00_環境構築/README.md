# S0 環境構築・ツールキット

本シリーズは、以降のすべてのシリーズが乗る土台を組み立てる。Python の計算環境と自作ライブラリ `bondlab` のレイヤー構成を整え、市場データを再現可能に取得・キャッシュするデータ層を用意する。派手な金融モデルは出てこないが、ここが正しくないと後続のカーブ構築・リスク計測・戦略バックテストがすべて狂う。

## 学ぶこと

- 債券クオンツで使う Python ライブラリを守備範囲で切り分ける（numpy/scipy で自作し、QuantLib で答え合わせをする方針）
- 自作ライブラリ `bondlab` のレイヤー構成（data → rates → bond → curve → …）と、依存を一方向にする理由
- 「自作関数には仕様表とテストを添える」開発フローを実際のコードで一往復する
- QuantLib の `Date` / `Calendar` / `Schedule` を操作し、各国カレンダーの営業日計算を行う
- 国債利回りデータの種類（パー利回り・ゼロレート・銘柄別利回り・価格）の区別と、ソースごとに何が手に入るか
- API キーを `.env` から読み、結果をキャッシュする再現可能なデータ層。鍵が無い環境では合成サンプルへフォールバックする

## なぜ重要か（実務での使われ方）

債券デスク・ファンドの収益源は、マーケットメイク（ビッド・オファースプレッド）、相対価値（rich/cheap・先物ベーシス）、キャリー、マクロ、運用会社の ALM に大別される（`docs/債券ファンドの業務.md` 参照）。そのどれもが、正しい日付処理と信頼できる市場データを暗黙の前提にしている。

- **日付・カレンダー**：利払日・受渡日・経過利子・レポの資金決済は営業日規約と各国祝日に依存する。一日ずれれば価格も損益も狂う。複数市場をまたぐ日米スプレッドや先物ベーシスでは、休場日の揃え方が指標そのものを変える。
- **市場データの区別**：年限グリッドのパー利回りはカーブ構築の入力になり、銘柄別利回りの「ばらつき」は相対価値デスクの稼ぎ場になる。両者を混同すると入力を誤る。
- **再現性**：同じ入力から同じ結果が返ることは、バックテストの信頼性・リスクレポートの監査可能性・モデルバリデーションの前提。`bondlab` を検証可能な小さな層の集合として育てる作法は、フロントのモデルを独立検証部門が承認する実務（SR 11-7 等）にそのまま通じる。

## 使う bondlab モジュール

- `bondlab.rates` — 複利規約の変換・割引係数（詳細は S1 で扱う。ここでは開発フローの題材として利用）
- `bondlab.data` — FRED 取得・キャッシュ・合成サンプル読み込みのデータアクセス層
- 外部：`QuantLib`（日付・カレンダー・スケジュール）、`numpy` / `pandas` / `matplotlib`

## 検証方針

- 自作実装は QuantLib を基準（ベンチマーク）として答え合わせする。numpy/scipy で組んだロジックの結果を独立実装と突き合わせる。
- notebook 内の `assert`（往復誤差・規約変換の不変性など）が、後でライブラリを直したときの回帰テストの下地になる。
- データ層は鍵の有無にかかわらず動くよう、キャッシュ専用モードと合成サンプルへのフォールバックを持たせ、CI・第三者検証をそのまま回せるようにする。

## 実行方法

```bash
pip install -e ".[dev]"
cp .env.example .env          # FRED_API_KEY は任意（無くても合成サンプルで動く）
export PYTHONPATH=$PWD
jupyter lab notebooks/S00_環境構築/
```

各 notebook は jupytext のペア `.ipynb` と `.py`（percent 形式）を両方コミットする。レビューは `.py` の差分で行い、`.ipynb` は実行済み出力つきで保持する。鍵が無い場合、`nb_0002` は `data/samples/` の合成サンプルで動作する。
